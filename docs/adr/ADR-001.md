# ADR-001: Isolamento Estrito de Protocolo (JIMI vs JT/T)

## Status
Aceito (implementado desde v6.2 do pushalarm)

## Contexto
O Jimi IoT Hub envia dados de alarme em dois protocolos distintos:
- **JIMI** (msgClass=0): utilizado pelos dispositivos da série JC400
- **JT/T 808** (msgClass=1): utilizado pelos dispositivos da série JC450/JC181

Cada protocolo possui:
- Códigos de alarme diferentes para o mesmo conceito (ex: colisão frontal = código 204 no JIMI, 264-1 no JT/T)
- Estruturas de payload diferentes (JIMI usa `alertType`, JT/T usa `alertType` + `standardAlarmValue` + `alarmSerialNo` + canais de vídeo)
- Mecanismos de resolução de nome diferentes (JIMI: tabela `alarm_types` com protocol='JIMI'; JT/T: mesma tabela com protocol='JTT' + bitmask decoding para código 256 + subtipos compostos 264-X, 265-X, 266-X)

Versões anteriores do handler misturavam a lógica dos dois protocolos, causando alarmes com nomes incorretos e campos JT/T sendo interpretados como JIMI.

## Decisão
Implementar isolamento estrito de protocolo em todo o ciclo de vida do alarme:

1. **Extração**: Campos específicos JT/T (`alarmSerialNo`, `signalDropChannel`, `standardAlarmValue`, `drivingAlarmFlag`, etc.) só são extraídos quando `msgClass == 1`
2. **Resolução de nome**: A consulta ao banco (`alarm_types`) sempre filtra por `protocol` ('JIMI' ou 'JTT'), nunca faz fallback entre protocolos
3. **Decodificação de bitmask**: Apenas para JT/T com `alertType=256`; o valor `standardAlarmValue` é decodificado bit-a-bit usando o mapa de 32 bits do JT/T 808-2019
4. **Subtipos compostos**: Códigos como `264-X` (ADAS), `265-X` (DMS), `266-X` (BSD) são resolvidos concatenando `alertType-subType` e consultando `alarm_types` com `protocol='JTT'`
5. **Fallback**: Se nenhum nome for encontrado, gera nome sintético com o protocolo explícito (`"Código 999 (JIMI)"` ou `"Código 999 (JTT)"`) — nunca assume um protocolo

## Consequências

**Positivas:**
- Alarmes JT/T e JIMI nunca são confundidos
- Expansão futura de códigos de alarme em qualquer protocolo não afeta o outro
- Nomes de alarme refletem precisamente a origem do protocolo
- Bitmask decoding captura todos os 32 bits de alarmes simultâneos

**Negativas:**
- A tabela `alarm_types` precisa conter entradas separadas para cada protocolo (código duplicado com protocol diferente é permitido pela unique key `uk_code_protocol`)
- A lógica de resolução de nome é mais complexa (3 caminhos: bitmask → subtipo composto → busca simples → fallback)
- Requer manutenção do mapa de bits JT/T caso o padrão 808 seja atualizado
